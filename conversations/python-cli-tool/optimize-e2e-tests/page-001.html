The current end to end tests, while functionally correct, incur extra run time (which is significant) vs what might be possible.
What I mean is at present each test case is testing some specific aspect of machine creation (tagging, dns, etc).
This is done by creating a new machine with a tag, then testing the tag is present, then destroying the machine.
The next test case then creates a new machine with dns, checks the dns, then destroys it. Etc.
I think it should be possible instead to : create a machine with tag, dns, etc. Have a set of test assertions
that the machine has the correct tag, dns etc, then destroy that one machine. It's important that each test assertion 
be reported as a separate pass/fail result. If all the assertions are done in a single "test", then when it fails it won't be easy to 
tell which aspect of the test failed. Usually test frameworks make it hard to have some common logic prior to several test cases, 
then some common cleanup code after they all run, but it's typically possible one way or another.
So although this will make the test code more complicated, in this case the benefit that the tests will run much quicker makes it worthwhile.
